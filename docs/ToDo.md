# デザインメモ: Aura Definition の「消費者」

## 前提（すでに実装済み）
- Aura mark は2種類に分かれる。
  - **binding**: `wantType:<型>|<section>/<key>` を指し、値を want のフィールドに貼る（従来の X マーク、mode set/endorse）。
  - **definition**: `<kind>:<name>` を指し、値（スカラーでもオブジェクトでも）がその名前の定義そのもの。want には適用されない。
- `AuraMark.Value` は `any`。定義はオブジェクトを持てる。署名（`by`）つき。
- 作る側UI: サイドバーの任意フィールドで X → インライン命名。kind はフィールドの subType から自動。`~/.mywant/characters.yaml` に保存。
- 読む側API: `ResolveAuraDefinition(kind, name, path)` / `AuraDefinitions(kind)`（engine/core/character_store.go）。**まだ誰も呼んでいない。**

## 本題: 定義は「作れる」が「使われて」いない
定義が実際に価値を生むのは、**誰かが名前で引いて挙動を変えたとき**。消費者がいないと、命名は死蔵される。以下は消費者の候補と設計の論点。

### 消費者パターン
1. **逆引き表示（座標→名前）**
   - location want の履歴/カードで、生の lat/lng を定義済みの place 名で表示する。
   - `AuraDefinitions("location_coordinate")` を引き、近接（radius内）なら名前に差し替え。
   - **遡って効く**のが利点（過去ログも名前で読める）。まず最小の消費者としてこれが良い。
   - 論点: 近接判定の距離しきい値をどこに持つか（定義側に radius / 消費側に固定値）。

2. **パラメータ補完（名前→値）**
   - transit の `from`/`to`（subType: station）等で、定義済みカタログをオートコンプリート候補に出す。
   - 「会社最寄り」を選ぶと station 値が入る。field-match/オートコンプリートの既存UIに candidates を足す形。
   - 論点: candidates のソースを定義に一本化するか、既存の datatype 候補とマージするか。

3. **結合キー（名前で束ねる）**
   - 位置ログ・天気・予定を「会社にいた」で束ねる。名前が共通キーになって初めて突き合わせ可能。
   - 消費者というより「名前解決を前段に噛ませた集計」。Thinker/OPA の入力を名前解決済みにする。

### 横断する論点
- **解決のタイミング**: 保存時に名前を焼き込む（速いが定義変更に追随しない）か、表示/実行時に毎回引く（重いが常に最新）か。定義は共有資産で書き換わりうるので、**実行時解決**を基本にすべき。
- **誰の定義を使うか**: 定義はキャラ署名つき。自分の定義が優先か、world 内の全定義をマージか。`ResolveAuraDefinition` は現状「誰かが持っていれば返す」= 先勝ち。多人数world では衝突しうる → 呼び出し側でキャラ優先度を指定できる版が要る。
- **名前空間 = kind**: いまは subType をそのまま kind にしている。station と city を別カタログにできる一方、型なしは "value" に集まる。カタログの粒度をどう設計するか（型ベースで十分か、明示的な namespace を持たせるか）。
- **一覧・編集UI**: 作った定義を見る場所がまだ無い。Characters ページに「この人が定義したカタログ」を出すのが自然。編集・削除・他人の定義の取り込み（fork）もここ。

### 最初の一手（推奨）
消費者1（座標→名前の逆引き表示）。理由: 経路が短く壊れにくい、遡って効く、既存の location カード内で完結、新エンドポイント不要。ここで「実行時解決」「キャラ優先度」「近接しきい値」の設計判断を実地で固めてから、2/3 に広げる。
